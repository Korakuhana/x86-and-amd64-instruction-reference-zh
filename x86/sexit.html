GETSEC[SEXIT] — 退出度量环境

操作码 指令 描述
NP 0F 37 (EAX=5) GETSEC[SEXIT] 退出度量环境。

描述¶

GETSEC[SEXIT] 指令发起退出由 GETSEC[SENTER] 建立的度量环境。在执行时将 EAX 设置为 5 来选择 GETSEC 的 SEXIT 叶函数。该指令叶函数向平台中的所有逻辑处理器发送消息，以发出退出度量环境的信号。

处理器对执行 GETSEC[SEXIT] 指令强制执行一些限制：

  • 除非处理器处于保护模式 (CR0.PE = 1) 且 CPL = 0 且 EFLAGS.VM = 0，否则不允许执行。
  • 处理器必须处于由先前的 GETSEC[SENTER] 指令启动的度量环境中，但不能仍处于已认证代码执行模式。
  • 为了避免模式之间潜在的互操作性冲突，如果处理器当前处于 SMM 或 VMX 操作中，则不允许执行此指令。
  • 为了确保一致地处理 SIPI 消息，执行 GETSEC[SEXIT] 指令的处理器还必须被指定为寄存器位 IA32_APIC_BASE.BSP（第 8 位）所定义的 BSP（自举处理器）。

如果不遵守上述条件，将导致处理器发出通用保护违规信号。

该指令启动一个将会合 RLP 与 ILP 同步的序列。然后，它清除指示处理器正在度量环境中运行的内部处理器标志。

作为对表示会合完成的消息的响应，所有 RLP 均使用在识别出 GETSEC[SEXIT] 时将要执行的指令重新开始执行。这适用于所有处理器状态，但以下情况除外：

  • 如果某个 RLP 执行了 HLT 并且在 GETSEC[SEXIT] 发起消息时处于此挂起（halt）状态，则在挂起状态下恢复执行。
  • 如果某个 RLP 正在执行 MWAIT，则由 GETSEC[SEXIT] 发起的消息会导致退出 MWAIT 状态，并落入下一条指令。
  • 如果某个 RLP 正在执行字符串指令的中间迭代，则处理器在识别出由 GETSEC[SEXIT] 发起的消息的位置恢复执行该字符串指令。
  • 如果某个 RLP 仍处于 SENTER 休眠状态（从未被 GETSEC[WAKEUP] 唤醒），则在首先清除自举处理器指示标志 (IA32_APIC_BASE.BSP) 和任何挂起的 SIPI 状态后，它将被送至等待-SIPI（wait-for-SIPI）状态。在这种情况下，此类 RLP 被初始化为与使用 INIT# 引脚进行软复位一致的架构状态。

在完成 GETSEC[SEXIT] 操作之前，ILP 和任何活动的 RLP 均解除对外部事件信号 INIT#、A20M、NMI# 和 SMI# 的响应屏蔽。该解除屏蔽是无条件执行的，以识别在 GETSEC[SENTER] 之后被屏蔽的引脚事件。A20M 的状态被解除屏蔽，因为在度量环境处于活动状态时无法识别 A20M 引脚。

成功退出度量环境后，ILP 会重新锁定支持 Intel® TXT 芯片组的私有配置空间。GETSEC[SEXIT] 不会影响任何 PCR 的内容。

在 ILP 完成 GETSEC[SEXIT] 时，执行将转到下一条指令。由于此指令不修改 EFLAGS 和调试寄存器状态，因此如果先前启用了挂起的陷阱条件，则可以自由发出信号。

单处理器平台中的操作¶

GETSEC[SEXIT]（仅限 ILP）：

IF (CR4.SMXE=0)

THEN #UD;

ELSE IF (in VMX non-root operation)

THEN VM Exit (reason=”GETSEC instruction”);

ELSE IF (GETSEC leaf unsupported)

THEN #UD;

ELSE IF ((in VMX root operation) or

(CR0.PE=0) or (CPL>0) or (EFLAGS.VM=1) or

(IA32_APIC_BASE.BSP=0) or

(TXT chipset not present) or

(SENTERFLAG=0) or (ACMODEFLAG=1) or (IN_SMM=1))

THEN #GP(0);

SignalTXTMsg(SEXIT);

DO

WHILE (no SignalSEXIT message);

TXT_SEXIT_MSG_EVENT (ILP & RLP):

Mask and clear SignalSEXIT event;

Clear MONITOR FSM;

Unmask SignalSENTER event;

IF (in VMX operation)

THEN TXT-SHUTDOWN(#IllegalEvent);

SignalTXTMsg(SEXITAck);

IF (logical processor is not ILP)

THEN GOTO RLP_SEXIT_ROUTINE;

(* ILP 等待所有逻辑处理器发出 ACK *)

DO

DONE := READ(LT.STS);

WHILE (NOT DONE);

SignalTXTMsg(SEXITContinue);

SignalTXTMsg(ClosePrivate);

SENTERFLAG := 0;

Unmask SMI, INIT, A20M, and NMI external pin events;

END;

RLP_SEXIT_ROUTINE（仅限 RLP）：

Wait for SignalSEXITContinue message;

Unmask SMI, INIT, A20M, and NMI external pin events;

IF (prior execution state = HLT)

THEN reenter HLT state;

IF (prior execution state = SENTER sleep)

THEN

IA32_APIC_BASE.BSP := 0;

Clear pending SIPI state;

Call INIT_PROCESSOR_STATE;

Unmask SIPI event;

GOTO WAIT-FOR-SIPI;

FI;

END;

受影响的标志位¶

ILP：无。

RLP：对于返回等待-SIPI 状态的 RLP，所有标志都被修改，否则无。

前缀的使用¶

LOCK 导致 #UD。

REP* 导致 #UD（包括 REPNE/REPNZ 和 REP/REPE/REPZ）。

操作数大小导致 #UD。

不允许使用 NP 66/F2/F3 前缀。

段重写被忽略。

地址大小被忽略。

REX 被忽略。

保护模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[SEXIT] 未报告支持 GETSEC[SEXIT]。
#GP(0) 如果 CR0.PE = 0 或 CPL > 0 或 EFLAGS.VM = 1。
如果处于 VMX 根操作中。
如果发起处理器未通过 MSR 位 IA32_APIC_BASE.BSP 指定。
如果不存在支持 Intel® TXT 的芯片组。
如果保护分区尚未激活或处理器已处于已认证代码模式。
如果处理器处于 SMM 中。

实地址模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[SEXIT] 未报告支持 GETSEC[SEXIT]。
#GP(0) GETSEC[SEXIT] 在实地址模式下无法识别。

虚拟-8086 模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[SEXIT] 未报告支持 GETSEC[SEXIT]。
#GP(0) GETSEC[SEXIT] 在虚拟-8086 模式下无法识别。

兼容模式异常¶

所有保护模式异常均适用。

64位模式异常¶

所有保护模式异常均适用。

#GP 如果 AC 代码模块不在低于 2^32 - 1 的物理地址中。

VM-Exit 条件¶

原因 (GETSEC) 如果处于 VMX 非根操作中。
