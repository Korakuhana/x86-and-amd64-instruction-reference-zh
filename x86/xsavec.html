XSAVEC — 使用紧凑方式保存处理器扩展状态

操作码/指令 Op/En 64/32位模式支持 CPUID 功能标志 描述
NP 0F C7 /4 XSAVEC mem M V/V XSAVEC 使用紧凑方式将由 EDX:EAX 指定的状态组件保存到 mem。
NP REX.W + 0F C7 /4 XSAVEC64 mem M V/N.E. XSAVEC 使用紧凑方式将由 EDX:EAX 指定的状态组件保存到 mem。

指令操作数编码¶

Op/En 操作数 1 操作数 2 操作数 3 操作数 4
M ModRM:r/m (w) N/A N/A N/A

描述¶

将处理器状态组件的全部或部分保存到由目的操作数指定的内存地址处的 XSAVE 区域。隐式的 EDX:EAX 寄存器对指定了 64 位指令掩码。保存的具体状态组件对应于请求功能位图 (RFBM) 中设置 of 位，该位图是 EDX:EAX 和 XCR0 的逻辑与。

XSAVE 区域的格式在《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4 节“XSAVE 区域”中详细介绍。像 FXRSTOR 和 FXSAVE 一样，用于 x87 状态的内存格式取决于 REX.W 前缀；参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.5.1 节“x87 状态”。

《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.10 节“XSAVEC 的操作”提供了关于 XSAVEC 指令操作的详细描述。以下条目提供了高级大纲：

  • XSAVEC 的执行类似于 XSAVE。XSAVEC 与 XSAVE 的不同之处在于它使用紧凑形式，并且它可以使用初始化优化。
  • 当且仅当 RFBM[i] = 1 且 XINUSE[i] = 1 时，XSAVEC 会保存状态组件 i。1（XINUSE 是一个位图，处理器通过它跟踪各种状态组件的状态。请参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.6 节“XSAVE 管理状态的处理器跟踪”。）
  • XSAVEC 不修改 XSAVE 区域传统区域的 511:464 字节（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.1 节“XSAVE 区域的传统区域”）。
  • XSAVEC 将 RFBM 和 XINUSE 的逻辑与写入 XSAVE 标头的 XSTATE_BV 字段。2,3（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.2 节“XSAVE 标头”。）XSAVEC 将 XCOMP_BV 字段的第 63 位设置为 1，并将该字段的 62:0 位设置为 RFBM[62:0]。除 XSTATE_BV 和 XCOMP_BV 字段外，XSAVEC 不写入 XSAVE 标头的任何其他部分。
  • XSAVEC 始终使用 XSAVE 区域扩展区域的紧凑格式（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.3 节“XSAVE 区域的扩展区域”）。

在 64 位或 32 位模式下，使用未对齐到 64 字节边界的目的操作数都会导致通用保护 (#GP) 异常。在 64 位模式下，RDX 和 RAX 的高 32 位被忽略。

1. 对于状态组件 1 (SSE) 存在特例。MXCSR 是 SSE 状态的一部分，但是即使 MXCSR 不具有其初始值 1F80H，XINUSE[1] 也可能为 0。在这种情况下，只要 RFBM[1] = 1，XSAVEC 就会保存 SSE 状态。

2. 与 XSAVE 和 XSAVEOPT 不同，XSAVEC 会清除 XSTATE_BV 字段中与 RFBM 中清除的位对应的位。

3. 对于状态组件 1 (SSE) 存在特例。MXCSR 是 SSE 状态的一部分，但是即使 MXCSR 不具有其初始值 1F80H，XINUSE[1] 也可能为 0。在这种情况下，只要 RFBM[1] = 1，XSAVEC 就会将 XSTATE_BV[1] 设置为 1。

操作¶

RFBM := XCR0 AND EDX:EAX;
                    /* bitwise logical AND */
TO_BE_SAVED := RFBM AND XINUSE;
                    /* bitwise logical AND */
If MXCSR ≠ 1F80H AND RFBM[1]
    TO_BE_SAVED[1] = 1;
FI;
IF TO_BE_SAVED[0] = 1
    THEN store x87 state into legacy region of XSAVE area;
FI;
IF TO_BE_SAVED[1] = 1
    THEN store SSE state into legacy region of XSAVE area; // this step saves the XMM registers, MXCSR, and MXCSR_MASK
FI;
NEXT_FEATURE_OFFSET = 576;
                    // Legacy area and XSAVE header consume 576 bytes
FOR i := 2 TO 62
    IF RFBM[i] = 1
        THEN
            IF TO_BE_SAVED[i]
                THEN save XSAVE state component i at offset NEXT_FEATURE_OFFSET from base of XSAVE area;
            FI;
            NEXT_FEATURE_OFFSET = NEXT_FEATURE_OFFSET + n (n enumerated by CPUID(EAX=0DH,ECX=i):EAX);
    FI;
ENDFOR;
XSTATE_BV field in XSAVE header := TO_BE_SAVED;
XCOMP_BV field in XSAVE header := RFBM OR 80000000_00000000H;

保护模式异常¶

#GP(0) 如果内存操作数有效地址超出 CS、DS、ES、FS 或 GS 段限制。
无论段如何，如果内存操作数未在 64 字节边界上对齐。
#SS(0) 如果内存操作数有效地址超出 SS 段限制。
#PF(错误码) 如果发生缺页异常。
#NM 如果 CR0.TS[bit 3] = 1。
#UD 如果 CPUID.01H:ECX.XSAVE[bit 26] = 0 或 CPUID.(EAX=0DH,ECX=1):EAX.XSAVEC[bit 1] = 0。
如果 CR4.OSXSAVE[bit 18] = 0。
如果使用了 LOCK 前缀。
#AC 如上所述，如果此异常被禁用，且内存操作数未对齐在 64 字节边界上，则会发出通用保护异常 (#GP)。如果启用了对齐检查异常 (#AC)（且 CPL 为 3），则不保证会发出 #AC，并且具体情况可能因实现而异，如下所示。在所有未发出 #AC 的实现中，会发出通用保护异常来代替它。此外，对齐检查的宽度也可能因实现而异。例如，对于特定的实现，可能会针对 2 字节未对齐发出对齐检查异常，而针对所有其他未对齐（4、8 或 16 字节未对齐）可能会发出通用保护异常。

如果操作数的任何部分超出 0 到 FFFFH 的有效地址空间。
#NM 如果 CR0.TS[bit 3] = 1。
#UD 如果 CPUID.01H:ECX.XSAVE[bit 26] = 0 或 CPUID.(EAX=0DH,ECX=1):EAX.XSAVEC[bit 1] = 0。
如果 CR4.OSXSAVE[bit 18] = 0。
如果使用了 LOCK 前缀。
