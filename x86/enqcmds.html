描述¶

ENQCMDS 指令允许系统软件向入队寄存器 (enqueue registers)写入命令，入队寄存器是通过内存映射 I/O (MMIO) 访问的特殊设备寄存器。

入队寄存器期望写入的数据具有图 3-16所给出的格式，并在“ENQCMD—Enqueue Command”一节中有所说明。

ENQCMDS 指令首先从源内存操作数中读取 64 字节的命令数据。这是一种普通加载，其可缓存性和内存顺序通常由内存类型隐式定义。源操作数不需要对齐，并且不保证所有 64 字节都是原子加载的。源操作数的位 30:20 必须为零。

ENQCMDS 格式化其源数据的方式与 ENQCMD 不同。具体来说，它将它们格式化为命令数据，格式如下：

  • Command[19:0] 获取源操作数的位 19:0。这 20 位用于传递进程地址空间标识符 (PASID)。
  • Command[30:20] 为零。
  • Command[511:31] 获取源操作数的位 511:31。位 31 传递特权标识（0 = 用户；1 = 管理员）。

然后该指令使用入队存储 (enqueue store)（定义见 ENQCMD）将此命令数据写入目标操作数。目标操作数的地址在通用寄存器中指定为 ES 段的偏移量（段不能被覆盖）。1 目标线性地址必须对齐到 64 字节边界。入队存储的操作忽略目标内存地址的内存类型。

1. 在 64 位模式下，寄存器操作数的宽度为 64 位（带有 67H 前缀时为 32 位）。在非 64 位模式下，当 CS.D = 1 时，宽度为 32 位（带有 67H 前缀时为 16 位）。在非 64 位模式下，当 CS.D = 0 时，宽度为 16 位（带有 67H 前缀时为 32 位）。

入队存储的排序与之前对 WB 或 WC 内存的存储（包括非临时存储）或 CLFLUSHOPT 或 CLWB 的执行（当应用于入队存储地址以外的地址时）无关。软件可以通过在入队存储之前执行内存屏障指令（如 SFENCE 或 MFENCE）来强制执行此类排序。

入队存储不会将数据写入缓存层次结构，也不会将任何数据提取到缓存层次结构中。入队存储的命令数据绝不会与写入同一地址的任何其他存储的数据合并。

与其他存储不同，入队存储会返回状态，ENQCMDS 指令将该状态加载到 RFLAGS 寄存器中的 ZF 标志中：

  • ZF = 0（成功）表示 64 字节的命令数据已原子方式写入到设备的入队寄存器并已被设备接受。（这并不保证设备已经执行了命令；它可能只是排队等候稍后执行。）
  • ZF = 1（重试）表示命令数据未被接受。如果目标地址是入队寄存器，但由于容量或其他临时原因导致命令未被接受，则返回此状态。如果目标地址不是入队寄存器（包括内存地址的情况），也会返回此状态；在这些情况下，存储会被丢弃，并且既不写入 MMIO 也不写入内存。

ENQCMDS 指令仅可在 CPL = 0 时执行。ENQCMDS 指令的可用性由 CPUID 功能标志 ENQCMD（CPUID.(EAX=07H, ECX=0H):ECX[位 29]）指示。

操作¶

DEST := SRC;

Intel C/C++ 编译器固有等效项¶

ENQCMDS int_enqcmds(void *dst, const void *src)

影响的标志寄存器¶

如果入队存储完成返回重试状态，则设置 ZF 标志；否则清除。清除所有其他标志。

SIMD 浮点异常¶

无。

保护模式异常¶

如果当前特权级别不为 0。
如果源操作数的位 30:20 不全为零。
#SS(0) 对于 SS 段中的非法地址。
#PF(错误代码) 如果发生页面错误。
#UD 如果 CPUID.07H.0H:ECX.ENQCMD[位 29] = 0。
如果使用了 LOCK 前缀。

#GP(0) ENQCMDS 指令在虚拟-8086 模式下不可识别。
