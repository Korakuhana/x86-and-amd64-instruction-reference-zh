GETSEC[ENTERACCS] — 执行已认证芯片组代码

操作码 指令 描述
NP 0F 37 (EAX = 2) GETSEC[ENTERACCS] 进入已认证代码执行模式。EBX 保存已认证代码模块物理基地址。ECX 保存已认证代码模块大小（字节）。

描述¶

GETSEC[ENTERACCS] 函数使用 Intel® TXT 平台芯片组的公钥加载、认证并执行已认证代码模块（Authenticated Code Module, ACM）。进入时，通过将 EAX 设置为 2 来选择 GETSEC 的 ENTERACCS 叶函数。

处理器对 GETSEC[ENTERACCS] 指令的执行实施了某些限制：

  • 除非处理器处于 CPL = 0 且 EFLAGS.VM = 0 的保护模式或 IA-32e 模式，否则不允许执行。
  • 处理器缓存必须可用且未被禁用，即 CR0.CD 和 CR0.NW 位必须为 0。
  • 对于包含多个逻辑处理器的处理器包，将检查 CR0.CD 以确保已启用的逻辑处理器之间的一致性。
  • 为了在使用中断 16 的数值异常报告中强制执行操作一致性，必须设置 CR0.NE。
  • 必须存在支持 Intel TXT 的芯片组，这是在复位后通过对上电配置功能字段进行采样传送给处理器的。
  • 处理器不能已经处于先前 GETSEC[ENTERACCS] 或 GETSEC[SENTER] 指令启动的已认证代码执行模式中，而之后没有使用 GETSEC[EXITAC] 退出。
  • 为避免模式之间潜在的操作冲突，如果处理器当前处于 SMM 或 VMX 操作中，则不允许执行此指令。
  • 为确保一致地处理 SIPI 消息，执行 GETSEC[ENTERACCS] 指令的处理器还必须被指定为 IA32_APIC_BASE.BSP（位 8）所定义的 BSP（引导处理器）。

如果不符合上述条件，将导致处理器触发通用保护异常。

在执行 ENTERACCS 叶函数之前，平台中的其他逻辑处理器（即 RLP）必须处于：

  • 在等待 SIPI 状态（由 INIT 断言发起，或对于非指定 BSP 处理器通过复位发起）中空闲，或者
  • 处于由发起逻辑处理器（ILP）通过 GETSEC[SENTER] 发起的 SENTER 睡眠状态。

如果同一包中的其他逻辑处理器未在这些状态之一中空闲，则执行 ENTERACCS 将触发通用保护异常。如果同一包的其他逻辑处理器未满足 CR0.CD = 0，也适用相同的要求和操作。

成功执行 ENTERACCS 会导致 ILP 进入已认证代码执行模式。在达到此点之前，处理器执行几项检查。这些检查包括：

  • 确立并检查要由处理器执行的指定已认证代码模块的位置和大小。
  • 抑制 ILP 对外部事件的响应：INIT、A20M、NMI 和 SMI。
  • 广播一条消息，以对其他处理器代理启用内存和 I/O 的保护。
  • 将指定的代码模块加载到已认证代码执行区域。
  • 隔离已认证代码执行区域的内容，防止外部代理对其状态进行进一步的修改。
  • 认证已认证代码模块。
  • 基于已认证代码模块头中包含的信息初始化发起逻辑处理器状态。
  • 解锁 Intel® TXT 芯片组私有配置空间和 TPM 权限级别（locality）3 空间。
  • 在定义的入口点开始执行已认证代码模块。

GETSEC[ENTERACCS] 函数在通用寄存器 EBX 和 ECX 中需要两个附加输入参数。EBX 保存已认证代码（AC）模块物理基地址（AC 模块必须驻留在物理地址空间 4 GBytes 以下），ECX 保存 AC 模块大小（以字节为单位）。物理基地址和大小用于从系统内存中检索代码模块，并将其加载到内部已认证代码执行区域中。检查基物理地址以验证它是否在 4096 字节的模边界上。验证大小是否为 64 的倍数，是否未超出内部已认证代码执行区域容量（由 GETSEC[CAPABILITIES] 报告），以及 AC 模块的最高地址是否未超出 32 位。错误条件会导致已认证代码执行启动中止，并触发通用保护异常。

作为对处理器硬件正确操作的完整性检查，执行 GETSEC[ENTERACCS] 还将检查所有机器检查状态寄存器（由 MSR IA32_MCi_STATUS 报告）的内容，以检测是否存在任何有效的不可纠正错误条件。此外，全局机器检查状态寄存器 IA32_MCG_STATUS 的 MCIP 位必须被清除，且 IERR 处理器包引脚（或其等效物）不得被断言，表示当前没有机器检查异常处理正在进行中。这些检查在开始加载已认证代码模块之前执行。此时，如果这些状态寄存器中存在任何未处理的有效不可纠正机器检查错误条件，将导致处理器触发通用保护违规。

ILP 屏蔽对外部信号 INIT#、A20M、NMI# 和 SMI# 断言的响应。此屏蔽保持活动状态，直到可选地被 GETSEC[EXITAC] 取消屏蔽（此定义的取消屏蔽行为假定 GETSEC[ENTERACCS] 不是由先前的 GETSEC[SENTER] 执行的）。此屏蔽控制的目的是防止暴露给可能不受已认证代码模块控制的现有外部事件处理程序。

ILP 设置一个内部标志以指示其已进入已认证代码执行模式。A20M 引脚的状态同样被屏蔽，并在内部强制为未断言状态，使得在已认证代码执行模式期间不识别任何外部断言。

为防止其他（逻辑）处理器干扰在已认证代码执行模式下运行的 ILP，来自其他处理器代理的内存（不包括隐式写回事务）访问和 I/O 将被阻止。此保护在 ILP 进入已认证代码执行模式时开始。只允许继续执行从 ILP 发起的内存 and I/O 事务。退出已认证代码执行模式是通过执行 GETSEC[EXITAC] 来完成的。内存和 I/O 活动的保护将一直有效，直到 ILP 执行 GETSEC[EXITAC]。

在使用 GETSEC[ENTERACCS] 或 GETSEC[SENTER] 启动已认证执行模块之前，必须首先将处理器的 MTRR（内存类型范围寄存器）初始化为将已认证 RAM 地址映射为 WB（写回）。如果不这样做，可能会影响处理器维持所加载的已认证代码模块隔离的能力。如果处理器检测到未满足此要求，它将在加载已认证代码模块期间触发带有错误代码的 Intel® TXT 复位条件。

虽然加载模块内的物理地址必须映射为 WB，但模块边界之外的位置的内存类型必须映射为由 GETSEC[PARAMETERS] 返回的受支持内存类型之一（或默认映射为 UC）。

为了符合 MTRR MSR 指定内存类型的最小粒度，已认证代码 RAM (ACRAM) 以 4096 字节粒度块的形式分配给处理器。如果 ECX 中指定的 AC 模块大小不是 4096 的倍数，则处理器将分配到下一个 4096 字节边界，以使用不确定数据映射为 ACRAM。此填充区域对已认证代码模块作为外部内存不可见，也不能依赖于用于填充填充区域的数据值。

成功完成 GETSEC[ENTERACCS] 后，处理器的架构状态会从已认证代码模块头部保存的内容中进行部分初始化。处理器 GDTR、CS 和 DS 选择器从已认证代码模块内的字段进行初始化。由于已认证代码模块必须是可重定位的，因此所有地址引用必须相对于 EBX 中的已认证代码模块基地址。处理器 GDTR 基值初始化为 AC 模块头部字段 GDTBasePtr + EBX 中保存的模块基地址，GDTR 界限设置为 GDTLimit 字段中的值。CS 选择器初始化为 AC 模块头部 SegSel 字段，而 DS 选择器初始化为 CS + 8。段描述符字段被隐式初始化为 BASE=0、LIMIT=FFFFFh、G=1、D=1、P=1、S=1、DS 的读/写访问权限，以及 CS 的执行/读访问权限。处理器开始执行已认证代码模块，EIP 设置为 AC 模块头部 EntryPoint 字段 + 模块基地址 (EBX)。检查用于初始化处理器状态的基于 AC 模块的字段的一致性，任何失败都会导致关机（shutdown）条件。

表 7-4 给出了处理器成功完成 GETSEC[ENTERACCS] 后寄存器状态初始化的摘要。进入已认证代码执行模式后，分页将被禁用。已认证代码模块被加载并使用物理地址最初执行。执行 GETSEC[ENTERACCS] 后，由系统软件确立新的（或恢复其先前的）分页环境以及适当的映射，以满足新的保护要求。EBP 初始化为已认证代码模块基物理地址，以便在已认证环境中进行初始执行。因此，鉴于已认证代码模块必须是位置无关的，已认证代码可以引用 EBP 进行基于相对地址的引用。

尚未由 GETSEC[ENTERACCS] 初始化的段相关处理器状态在使用前需要进行适当的初始化。由于已经建立了新的 GDT 上下文，因此段选择器寄存器 ES、SS、FS、GS、TR 和 LDTR 中保存的先前状态值可能无效。

MSR IA32_EFER 也会作为由 ENTERACCS 初始化的处理器状态的一部分而被无条件清除。由于进入已认证代码执行模式时分页被禁用，因此必须重新建立新的分页环境，以便在已认证代码执行模式下运行时建立 IA-32e 模式。

调试异常和与陷入相关的信号也在 GETSEC[ENTERACCS] 中被禁用。这是通过复位 DR7、EFLAG 中的 TF 和 MSR IA32_DEBUGCTL 来实现的。在进入已认证代码执行模式后，一旦正确初始化了支持的异常处理程序、描述符表和调试寄存器，即可自由地重新启用这些调试功能。此外，进入此模式后，任何挂起的单步陷入条件都将被清除。

在执行 ENTERACCS 期间，与性能相关的计数器和计数器控制寄存器将被清除。这意味着在执行 ENTERACCS 时，任何处于活动状态的性能计数器都将被禁用。要重新激活处理器性能计数器，必须重新初始化并重新启用此状态。

IA32_MISC_ENABLE MSR 在进入已认证执行模式时被初始化。此 MSR 的某些位会被保留，因为保留这些位可能对于维持先前确立的平台设置至关重要（请参阅表 7-5 的力注）。其余位被清除，目的是为已认证代码模块的执行建立更一致的环境。初始化此 MSR 的影响之一是由 MONITOR 指令确立的任何先前条件都将被清除。

为了支持可能在执行 GETSEC[ENTERACCS] 之前返回到处理器架构状态，在指令完成时，某些关键的处理器状态会被捕获并存储在通用寄存器中。[E|R]BX 保存将在 GETSEC[ENTERACCS] 之后执行的下一条指令的有效地址（[E|R]IP），ECX[15:0] 保存 CS 选择器值，ECX[31:16] 保存 GDTR 界限字段，而 [E|R]DX 保存 GDTR 基址字段。随后的已认证代码可以保存这些寄存器的内容，以便在需要时，在通过 GETSEC[EXITAC] 退出已认证代码执行模式之前手动恢复此状态。有关退出已认证代码执行模式后的处理器状态，请参见 GETSEC[SEXIT] 的描述。

1. 初始化的 IA32_MISC_ENABLE 字段的数量可能会因处理器实现而异。

2. ENTERACCS（和 SENTER）初始化处理器热调制的防过热保护状态，使其至少启用最低级别。如果在执行这些 GETSEC 叶函数之一时热调制已被启用，则热调制控制设置不会发生改变。如果禁用了热调制，则会通过执行这些 GETSEC 叶函数设置热调制控制位 3 来启用它。

在进入已认证代码执行模式后，在能够处理任何异常或外部中断 INTR 和 NMI 之前，IDTR 还将需要重新加载新的 IDT 上下文。由于在已认证代码执行模式完成时（以 EXITAC 终止）会重新启用外部中断，因此建议在此点之前建立新的 IDT 上下文。在建立这种新的 IDT 上下文之前，程序员必须注意不要执行 INT n 指令或导致产生异常或陷入信号的任何其他操作。

在完成 GETSEC[ENTERACCS] 指令之前且成功认证 AC 模块之后，Intel TXT 芯片组的私有配置空间被解锁。只有已认证代码模块能访问此通常受限制的芯片组状态，以确保平台安全。

一旦已认证代码模块在完成 GETSEC[ENTERACCS] 时启动，它就可以通过设置 EFLAGS.IF 来启用中断，并通过执行 IRET 来启用 NMI。这假定它已通过初始化 IDT、GDT 和相应的中断处理代码重新建立了中断处理支持。

IF (CR4.SMXE=0)

THEN #UD;

ELSIF (in VMX non-root operation)

ELSIF (GETSEC leaf unsupported)

(TXT chipset not present) or

(ACMODEFLAG=1) or (IN_SMM=1))

(secondary thread(s) not in SENTER sleep state)

THEN #GP(0);

IF (KEYHASH ≠ CSKEYHASH)

影响的标志寄存器¶

清除所有标志。

前缀的使用¶

LOCK 导致 #UD。

REP* 导致 #UD（包括 REPNE/REPNZ 和 REP/REPE/REPZ）。

操作数大小 导致 #UD。

不允许 NP 66/F2/F3 前缀。

段覆盖被忽略。

地址大小被忽略。

REX 被忽略。

保护模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[CAPABILITIES] 未报告支持 GETSEC[ENTERACCS]。
#GP(0) 如果 CR0.CD = 1 或 CR0.NW = 1 或 CR0.NE = 0 或 CR0.PE = 0 或 CPL > 0 或 EFLAGS.VM = 1。
如果不存在支持 Intel® TXT 的芯片组。
如果处于 VMX root 操作中。
如果发起处理器未通过 MSR 位 IA32_APIC_BASE.BSP 被指定为引导处理器。
如果处理器已处于已认证代码执行模式。
如果处理器处于 SMM 模式。
如果 IA32_MC[I]_STATUS 中记录了有效的不可纠正机器检查错误。
如果已认证代码基地址不在 4096 字节边界上。
如果已认证代码大小 > 处理器内部已认证代码区域容量。
如果已认证代码大小不是 64 的模。
如果同包的其他已启用逻辑处理器 CR0.CD = 1。
如果同包的其他已启用逻辑处理器未处于 wait-for-SIPI 或 SENTER 睡眠状态。

实地址模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[CAPABILITIES] 未报告支持 GETSEC[ENTERACCS]。
#GP(0) GETSEC[ENTERACCS] 在实地址模式下不可识别。

虚拟-8086 模式异常¶

#UD 如果 CR4.SMXE = 0。
如果 GETSEC[CAPABILITIES] 未报告支持 GETSEC[ENTERACCS]。
#GP(0) GETSEC[ENTERACCS] 在虚拟-8086 模式下不可识别。

兼容模式异常¶

所有保护模式异常均适用。

#GP 如果 AC 代码模块没有驻留在 2^32 - 1 以下的物理地址空间。

64位模式异常¶

所有保护模式异常均适用。

#GP 如果 AC 代码模块没有驻留在 2^32 - 1 以下的物理地址空间。

VM-exit 条件¶

如果在 VMX 非 root 操作中，退出原因（GETSEC）。
