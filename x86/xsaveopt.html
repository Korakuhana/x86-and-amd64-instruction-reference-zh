XSAVEOPT — 优化保存处理器扩展状态

操作码/指令 Op/En 64/32位模式支持 CPUID 功能标志 描述
NP 0F AE /6 XSAVEOPT mem M V/V XSAVEOPT 将由 EDX:EAX 指定的状态组件保存到 mem，如果可能则进行优化。
NP REX.W + 0F AE /6 XSAVEOPT64 mem M V/V XSAVEOPT 将由 EDX:EAX 指定的状态组件保存到 mem，如果可能则进行优化。

指令操作数编码¶

Op/En 操作数 1 操作数 2 操作数 3 操作数 4
M ModRM:r/m (r, w) N/A N/A N/A

描述¶

将处理器状态组件的全部或部分保存到由目的操作数指定的内存地址处的 XSAVE 区域。隐式的 EDX:EAX 寄存器对指定了 64 位指令掩码。保存的具体状态组件对应于请求功能位图 (RFBM) 中设置的位，该位图是 EDX:EAX 和 XCR0 的逻辑与。

XSAVE 区域的格式在《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4 节“XSAVE 区域”中详细介绍。像 FXRSTOR 和 FXSAVE 一样，用于 x87 状态的内存格式取决于 REX.W 前缀；参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.5.1 节“x87 状态”。

《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.9 节“XSAVEOPT 的操作”提供了关于 XSAVEOPT 指令操作的详细描述。以下条目提供了高级大纲：

  • XSAVEOPT 的执行类似于 XSAVE。XSAVEOPT 与 XSAVE 的不同之处在于它可以使用初始化优化和修改优化。XSAVEOPT 的性能将等于或优于 XSAVE。
  • 仅当 RFBM[i] = 1 且 XINUSE[i] = 1 时，XSAVEOPT 才会保存状态组件 i。1（XINUSE 是一个位图，处理器通过它跟踪各种状态组件的状态。请参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.6 节“XSAVE 管理状态的处理器跟踪”。）即使这两位都为 1，XSAVEOPT 也可能会在以下情况下优化且不保存状态组件 i：(1) 自上次执行 XRSTOR 或 XRSTORS 以来，状态组件 i 未被修改；并且 (2) 根据内部值 XRSTOR_INFO 的确定，本次执行的 XSAVES（注：此处原文为 XSAVES，实指 XSAVEOPT）与上次执行 of XRSTOR 或 XRSTORS 对应（参见下面的操作部分）。
  • XSAVEOPT 不修改 XSAVE 区域传统区域的 511:464 字节（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.1 节“XSAVE 区域的传统区域”）。
  • XSAVEOPT 读取 XSAVE 标头的 XSTATE_BV 字段（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.2 节“XSAVE 标头”），并将修改后的值写回内存，如下所示。如果 RFBM[i] = 1，XSAVEOPT 用 XINUSE[i] 的值写入 XSTATE_BV[i]。如果 RFBM[i] = 0，XSAVEOPT 用它从内存中读取的值写入 XSTATE_BV[i]（它不修改该位）。除 XSTATE_BV 字段外，XSAVEOPT 不写入 XSAVE 标头的任何其他部分。
  • XSAVEOPT 始终使用 XSAVE 区域扩展区域的标准格式（参见《Intel® 64 和 IA-32 架构软件开发人员手册》卷 1 第 13.4.3 节“XSAVE 区域的扩展区域”）。

在 64 位或 32 位模式下，使用未对齐到 64 字节边界的目的操作数都会导致通用保护 (#GP) 异常。在 64 位模式下，RDX 和 RAX 的高 32 位被忽略。

1. 对于属于状态组件 1 — SSE 的 MXCSR 和 MXCSR_MASK，存在特例。如果 RFBM[1] = 1 或 RFBM[2] = 1，无论 XINUSE 的值如何，XSAVEOPT 都会将这些保存到内存中。

操作¶

RFBM := XCR0 AND EDX:EAX; /* bitwise logical AND */
OLD_BV := XSTATE_BV field from XSAVE header;
TO_BE_SAVED := RFBM AND XINUSE;
IF in VMX non-root operation
    THEN VMXNR := 1;
    ELSE VMXNR := 0;
FI;
LAXA := linear address of XSAVE area;
IF XRSTOR_INFO = CPL,VMXNR,LAXA,00000000_00000000H
    THEN TO_BE_SAVED := TO_BE_SAVED AND XMODIFIED;
FI;
IF TO_BE_SAVED[0] = 1
    THEN store x87 state into legacy region of XSAVE area;
FI;
IF TO_BE_SAVED[1]
    THEN store XMM registers into legacy region of XSAVE area; // this step does not save MXCSR or MXCSR_MASK
FI;
IF RFBM[1] = 1 or RFBM[2] = 1
    THEN store MXCSR and MXCSR_MASK into legacy region of XSAVE area;
FI;
FOR i := 2 TO 62
    IF TO_BE_SAVED[i] = 1
        THEN save XSAVE state component i at offset n from base of XSAVE area (n enumerated by CPUID(EAX=0DH,ECX=i):EBX);
    FI;
ENDFOR;
XSTATE_BV field in XSAVE header := (OLD_BV AND NOT RFBM) OR (XINUSE AND RFBM);

保护模式异常¶

#GP(0) 如果内存操作数有效地址超出 CS、DS、ES、FS 或 GS 段限制。
无论段如何，如果内存操作数未在 64 字节边界上对齐。
#SS(0) 如果内存操作数有效地址超出 SS 段限制。
#PF(错误码) 如果发生缺页异常。
#NM 如果 CR0.TS[bit 3] = 1。
#UD 如果 CPUID.01H:ECX.XSAVE[bit 26] = 0 或 CPUID.(EAX=0DH,ECX=1):EAX.XSAVEOPT[bit 0] = 0。
如果 CR4.OSXSAVE[bit 18] = 0。
如果使用了 LOCK 前缀。
#AC 如上所述，如果此异常被禁用，且内存操作数未对齐在 64 字节边界上，则会发出通用保护异常 (#GP)。如果启用了对齐检查异常 (#AC)（且 CPL 为 3），则不保证会发出 #AC，并且具体情况可能因实现而异，如下所示。在所有未发出 #AC 的实现中，会发出通用保护异常来代替它。此外，对齐检查的宽度也可能因实现而异。例如，对于特定的实现，可能会针对 2 字节未对齐发出对齐检查异常，而针对所有其他未对齐（4、8 或 16 字节未对齐）可能会发出通用保护异常。
