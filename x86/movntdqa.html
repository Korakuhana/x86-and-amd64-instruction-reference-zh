描述¶

如果内存源是 WC（写合并）内存类型，MOVNTDQA 使用非临时提示将双四字从源操作数（第二个操作数）加载到目的操作数（第一个操作数）。对于 WC 内存类型，非临时提示可以通过将等同于对齐缓存行的内容加载到临时内部缓冲区中，而不将该数据填充到缓存中来实现。缓存中的任何具有别名内存类型的行将被探测（snooped）并清除（flushed）。如果数据可用，随后对 WC 缓存行未读取部分的 MOVNTDQA 读取将从临时内部缓冲区接收数据。处理器可能随时因任何原因刷新临时内部缓冲区，例如：

  • 除了引用已驻留在临时内部缓冲区中的内存的 MOVNTDQA 之外的加载操作。
  • 对已驻留在临时内部缓冲区中的内存的非 WC 引用。
  • 对单个临时内部缓冲区的交替读写。
  • 流式行中特定 16 字节项的重复 (V)MOVNTDQA 加载。
  • 某些微架构条件，包括资源短缺、检测到错误预测条件以及各种故障条件。

非临时提示是通过在从内存读取数据时使用写合并 (WC) 内存类型协议来实现的。使用此协议，处理器不会将数据读取到缓存层次结构中，也不会从内存中将相应的缓存行提取到缓存层次结构中。如果为非临时读取指定的内存地址不是 WC 内存区域，则被读取区域的内存类型可以覆盖非临时提示。有关非临时读取和写入的信息，请参阅《Intel® 64 和 IA-32 架构软件开发人员手册》第 3A 卷第 10 章中的“时间数据与非时间数据的缓存”。

因为 WC 协议使用弱有序内存一致性模型，所以如果多个处理器可能会对引用的内存位置使用不同的内存类型，或者为了将处理器的读取与系统中其他代理的写入同步，应将使用 MFENCE 指令实现的屏障操作与 MOVNTDQA 指令结合使用。处理器对流式加载提示的实现不会覆盖有效内存类型，但提示的实现取决于处理器。例如，处理器实现可以选择忽略提示，并在任何内存类型下将该指令作为普通 MOVDQA 处理。或者，另一种实现可以优化由 MOVNTDQA 在 WB 内存类型上生成的缓存读取，以减少缓存驱逐。

128 位 (V)MOVNTDQA 地址必须 16 字节对齐，否则该指令将导致 #GP 异常。

256 位 VMOVNTDQA 地址必须 32 字节对齐，否则该指令将导致 #GP 异常。

512 位 VMOVNTDQA 地址必须 64 字节对齐，否则该指令将导致 #GP 异常。

操作¶

MOVNTDQA (128bit- Legacy SSE Form)¶

DEST := SRC
DEST[MAXVL-1:128] (Unmodified)

VMOVNTDQA (VEX.128 and EVEX.128 Encoded Form)¶

DEST := SRC
DEST[MAXVL-1:128] := 0

VMOVNTDQA (VEX.256 and EVEX.256 Encoded Forms)¶

DEST[255:0] := SRC[255:0]
DEST[MAXVL-1:256] := 0

VMOVNTDQA (EVEX.512 Encoded Form)¶

DEST[511:0] := SRC[511:0]
DEST[MAXVL-1:512] := 0
