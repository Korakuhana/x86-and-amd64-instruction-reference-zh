FXSAVE — 保存 x87 FPU、MMX 技术和 SSE 状态

操作码/指令 Op/En 64位模式 兼容/传统模式 描述
NP 0F AE /0 FXSAVE m512byte M Valid Valid 将 x87 FPU、MMX、XMM 和 MXCSR 寄存器状态保存到 m512byte。
NP REX.W + 0F AE /0 FXSAVE64 m512byte M Valid N.E. 将 x87 FPU、MMX、XMM 和 MXCSR 寄存器状态保存到 m512byte。

指令操作数编码¶

Op/En 操作数 1 操作数 2 操作数 3 操作数 4
M ModRM:r/m (w) N/A N/A N/A

描述¶

将 x87 FPU、MMX 技术、XMM 和 MXCSR 寄存器的当前状态保存到目的操作数中指定的 512 字节内存位置。512 字节区域的内容布局取决于处理器是运行在非 64 位运行模式下，还是运行在 IA-32e 模式的 64 位子模式下。

字节 464:511 可供软件使用。处理器不会写入 FXSAVE 区域的 464:511 字节。

首先介绍 FXSAVE 在非 64 位模式下的操作。

非 64 位模式操作¶

表 3-43 显示了处理器在传统模式下运行时内存中状态信息的布局。

目的操作数包含内存映像的第一个字节，它必须对齐在 16 字节边界上。未对齐的目的操作数将导致产生一般保护 (#GP) 异常（或在某些情况下产生对齐检查异常 [#AC]）。

当操作系统需要进行上下文切换或异常处理程序需要保存并检查 x87 FPU、MMX 技术和/或 XMM 和 MXCSR 寄存器的当前状态时，使用 FXSAVE 指令。

表 3-43 中的字段定义在表 3-44 中。

FXSAVE 指令在 FTW 字段中保存缩简版本的 x87 FPU 标记字（与保存完整标记字的 FSAVE 指令不同）。标记信息是按照物理寄存器顺序（R0 至 R7）保存的，而不是按照栈顶 (TOS) 顺序。然而，使用 FXSAVE 指令，每个标记只保存一个位（1 表示有效，0 表示空）。例如，假设标记字当前设置如下：

R7 R6 R5 R4 R3 R2 R1 R0

11 xx xx xx 11 11 11 11

这里，11B 表示空栈元素，“xx”表示有效 (00B)、零 (01B) 或特殊 (10B)。

对于本例，FXSAVE 指令仅保存以下 8 位信息：

R7 R6 R5 R4 R3 R2 R1 R0

01110000

这里，对于任何有效、零或特殊标记，保存 1，对于任何空标记，保存 0。

FXSAVE 指令的操作与 FSAVE 指令的操作不同之处如下：

  • FXSAVE 指令不检查待处理的未屏蔽浮点异常。（在这方面，FXSAVE 操作类似于 FNSAVE 指令的操作）。
  • 在 FXSAVE 指令保存了 x87 FPU、MMX 技术、XMM 和 MXCSR 寄存器的状态之后，处理器保留寄存器的内容。由于这种行为，应用程序不能使用 FXSAVE 指令将“干净的” x87 FPU 状态传递给过程，因为它保留了当前状态。要清理 x87 FPU 状态，应用程序必须在 FXSAVE 指令之后显式执行 FINIT 指令以重新初始化 x87 FPU 状态。
  • 无论当前的寻址模式（32 位或 16 位）和运行模式（保护模式、实地址模式或系统管理模式）如何，使用 FXSAVE 指令保存的内存映像的格式都是相同的。这种行为与 FSAVE 指令不同，后者的内存映像格式根据寻址模式和运行模式而有所不同。由于映像格式不同，使用 FXSAVE 指令保存的内存映像无法使用 FRSTOR 指令正确恢复，同样，使用 FSAVE 指令保存的状态也无法使用 FXRSTOR 指令正确恢复。

可以通过 FTW 有效位和存储的 80 位浮点数据（假设存储的数据不是 MMX 技术寄存器的内容），使用表 3-45 重新创建 FTW 的 FSAVE 格式。

J 位定义为尾数中十进制小数点左侧的 1 位二进制整数。M 位定义为尾数小数部分的最有效位（即紧靠十进制小数点右侧的位）。

当 M 位是尾数小数部分的最有效位时，如果小数全为 0，则它必须为 0。

IA-32e 模式操作¶

在 IA-32e 模式的兼容子模式中，传统 SSE 寄存器 XMM0 至 XMM7 按照传统 FXSAVE 图进行保存。在 64 位模式下，所有的 SSE 寄存器 XMM0 至 XMM15 都会被保存。此外，在 64 位模式下，FXSAVE 图有两种不同的布局，分别对应于 FXSAVE64（需要 REX.W=1）和 FXSAVE（REX.W=0）。在 FXSAVE64 图（表 3-46）中，FPU IP 和 FPU DP 指针为 64 位宽。在 64 位模式的 FXSAVE 图（表 3-47）中，FPU IP 和 FPU DP 指针为 32 位。

操作¶

IF 64-Bit Mode
    THEN
        IF REX.W = 1
            THEN
                DEST := Save64BitPromotedFxsave(x87 FPU, MMX, XMM15-XMM0,
                MXCSR);
            ELSE
                DEST := Save64BitDefaultFxsave(x87 FPU, MMX, XMM15-XMM0, MXCSR);
        FI;
    ELSE
        DEST := SaveLegacyFxsave(x87 FPU, MMX, XMM7-XMM0, MXCSR);
FI;

保护模式异常¶

#GP(0) 如果 CS、DS、ES、FS 或 GS 段中的内存操作数有效地址非法。
如果内存操作数未对齐在 16 字节边界上，无论段如何。（请参阅下文对齐检查异常 [#AC] 的描述。）
#SS(0) 如果 SS 段中的地址非法。
#PF(错误码) 如果发生缺页异常。
#NM 如果 CR0.TS[bit 3] = 1。
如果 CR0.EM[bit 2] = 1。
#UD 如果 CPUID.01H:EDX.FXSR[bit 24] = 0。
#UD 如果使用了 LOCK 前缀。
#AC 如果禁用此异常，且内存操作数未对齐在 16 字节边界上，则如上所述发出一般保护异常 (#GP) 信号。如果启用了对齐检查异常 (#AC)（且 CPL 为 3），则不保证发出 #AC 信号，并且可能因实现而异，如下所示。在不发出 #AC 信号的所有实现中，将发出一般保护异常信号来代替。此外，对齐检查的宽度也可能因实现而异。例如，对于给定的实现，可能会对 2 字节未对齐发出对齐检查异常信号，而对所有其他未对齐（4、8 或 16 字节未对齐）可能会发出一般保护异常信号。

实地址模式异常¶

#GP 如果内存操作数未对齐在 16 字节边界上，无论段如何。
如果操作数的任何部分位于从 0 到 FFFFH 的有效地址空间之外。
#NM 如果 CR0.TS[bit 3] = 1。
如果 CR0.EM[bit 2] = 1。
#UD 如果 CPUID.01H:EDX.FXSR[bit 24] = 0。
如果使用了 LOCK 前缀。

虚拟-8086 模式异常¶

与实地址模式下的异常相同。

#PF(错误码) 如果发生缺页异常。
#AC 如果内存引用未对齐。
#UD 如果使用了 LOCK 前缀。

实现注意事项¶

当一般保护异常 (#GP) 和缺页异常 (#PF) 都在指令边界上发生时，处理器发出信号的顺序参见《Intel® 64 和 IA-32 架构软件开发人员手册》第 3B 卷表 5-2。该顺序在不同的处理器实现中对 FXSAVE 可能有所不同。
